Features

Features of our Jupiter System

List of the global features of our implementation:

  • Jupiter Server using Node + Socket.io
    • Easily scalable main server using Socket.io to dynamically communicate with the clients.
  • Jupiter Client using HTML + JS
    • Client interfaces with an engine able to generate and receive operations, only using the last web standards.
  • Real-Time Synchronization of the Server and all the Clients
    • Based on the lightweight communication protocol
  • Handling of any kind of collisions
    • Thanks to the implementation of the xform method and to the closure of the set of operations under this method, every potential kind of collision is covered. (see Operations & Transformation Rules)
  • MVC Structure
    • Using Express server-side and HTML5 Boilerplate client-side, a clear architecture has been chosen.
  • Protection of the content behind sessions (only authorized users can access)
    • The main Web pages and the Websockets are both protected by sessions.
  • Persistence using MongoDB
    • To continue with the new hype techno, this noSQL DB has been chosen to persist the state of the Jupiter System (Users, Notes, Operations).
  • Post-Crash Recovery
    • Thanks to the persistence, the server can crash/stop, and then restart in the same state as previously.
  • 1 User, n Clients
    • A User is identified by a username and a password, and can run simultaneously several client nodes without interferences.
  • Awareness of other Users and their activity
    • Each User can see which other User was responsible for a given operation.
  • Logs
    • Logs are kept server-side (detailed stuff is also printed in the client-side console).
  • REST API
    • Next to the HTML/WebSocket server, a REST server can be lauched, with some basic CRUD services.
  • Web Workers
    • For heavy client-side instructions, Web-workers are used, to make full use of the new power of JS.

Features of our demo notepal

List of some features specific for our small demo:

  • Users Profile
    • Creation + Authentification for unique Users
    • Passwords are strongly encrypted (bcrypt)
  • Jupiter Operations to manipulate sticky notes, ie possibility to:
    • Add notes
    • Delete notes
    • Edit the content of the notes
    • Drag'n drop the notes
  • Inteligent Notes Stacking
    • Notes are stacked from the last one updated to the most recent
  • Who's doing what?
    • Updated list of Users and their number of clients visible on the GUI
    • Last and current activity of each client visible on the GUI (small colored dot in the corner of the currently edited note)
  • CSS3+HTML5 prettyfication
    • Just to continue playing with hype stuff.

Limitations

  • Security
    • No HTTPS yet
      • Solution: SSL protection has been implemented but not activated yet, a conform certificate must be created first.
    • No condition on the users passwords
      • Solution: Improve the form
  • Reliability
    • If the server crashes for any reason, the current and following users operations won't be taken into account.
      • Solution: Implement a system of acknowledgments for the Jupiter communications, to avoid losing messages
    • If a client loses its connection, it will also lose its synchronization (its operation won't be taken into account)
      • Solution: see above (store all non-acknowledged messages and send them once the connection restaured)
  • REST API
    • Not protected yet
      • Solution: The current implementation was just a demo. Protecting private methods behing sessions wouldn't be complicated.
    • Not complete
      • Solution: The current implementation was just a demo. Adding more useful features (retrieve notes as files, administrate the system, ...) will be done in time.
  • Text Edition Handling
    • Currently, the Google's library diff-match-patch is used to find differences between the previous content of a note and the new one, after an input has been noticed. However, it is a heavy solution (though processed by Web-workers), and a better method can probably be found.
      • Solution: Watch for any kind of events (keyboard, mouse, ...) and cover all the potential cases?
